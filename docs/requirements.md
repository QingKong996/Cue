# 需求分析

## 产品目标

Cue 的第一版目标不是做完整会议助手，而是证明一个核心体验：用户在会议中被直接点名时，能够立刻看到足够短、足够有用的上下文提示，从而自然接住对话。

## 目标用户

- 开会时经常并行处理任务，但又担心突然被点名的人。
- 远程会议中容易错过话题转折的人。
- 使用第二语言开会、需要快速恢复上下文的人。
- ADHD、听觉处理障碍或容易短暂断线的人。

## 用户流程

1. 用户打开 Cue Web 页面。
2. 用户输入自己的主称呼和可选别名，例如姓名、英文名、昵称，并标注不同优先级。
3. 用户选择音频来源：麦克风或共享标签页/屏幕音频。
4. Cue 进入静默监听状态。
5. 页面显示极低干扰的运行状态，不主动刷存在感。
6. 当检测到用户被直接点名时，Cue 显示一句上下文提示。
7. 用户读完后，提示自动淡出或手动关闭。

## 功能需求

- 音频授权：引导用户授权麦克风或屏幕/标签页共享。
- 实时转写：接入字节跳动/火山引擎豆包流式语音识别模型 2.0，将音频转换为连续文本。
- 模型接入：Demo 阶段统一使用 DeepSeek API。
- 用户称呼配置：支持主称呼、明确昵称、弱别名，并允许为不同称呼设置不同优先级。
- 称呼候选生成：使用 `deepseek-v4-pro` 生成可能称呼、弱别名和角色称呼。
- 点名检测：识别用户是否被直接提问。
- 弱称呼处理：例如用户叫“张三”时，“小张”只能作为低置信候选，不能直接等同于“张三”。
- 语义判定：结合当前句子、最近上下文、用户角色和称呼置信度判断是否真的在问用户。
- 上下文窗口：维护最近会议内容。
- CueFrame 维护：使用 `deepseek-v4-pro` 按分钟级节奏，把新增上下文和上一版 CueFrame 交给模型，增量维护较具体的上下文框架。
- 展示门控：点名候选出现后做语义检测，检测为真时基于 CueFrame 生成正式 Cue，检测为假时保持静默。
- 正式 Cue 生成：使用 `deepseek-v4-flash` 输出一句短提示，必要时扩展为最多 2-3 个要点。
- 提示展示：以浮层或固定角落卡片出现，不遮挡主要工作区。
- 系统通知：正式 Cue 生成后，同时通过浏览器系统通知提醒用户。
- 静默模式：未被点名时不展示详细内容。

## 非功能需求

- 延迟：从转写文本出现到提示展示，目标控制在 2 秒内。
- 可读性：提示必须能在一眼内扫完。
- 隐私：默认不保存会议内容，页面刷新后清空会话。
- 权限透明：明确说明当前使用的是麦克风还是共享源音频。
- 通知权限：首次启用时请求浏览器通知权限；用户拒绝时降级为页面内提示。
- 浏览器优先级：先支持最新版 Chrome，其他浏览器后续评估。
- 失败可恢复：音频权限失败、共享源没有音频轨道、转写失败时，页面给出清晰状态。

## 原型边界

第一版只做 Web 原型，不做：

- 会后纪要。
- 会议日历集成。
- 团队空间。
- 历史记录。
- 多人账号系统。
- 任意桌面应用的系统音频捕获。
- 自动加入会议。
- 拼音化或同音近音召回。

## 演示建议

演示时使用 Chrome 打开 Cue，再播放一段网页会议音频，或打开网页版会议并共享标签页音频。这样最能体现 Web-first 路线，也能避开纯浏览器无法稳定捕获系统全局音频的问题。
