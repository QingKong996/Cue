# 点名检测设计

## 核心原则

Cue 的点名检测不能依赖单纯字符匹配。字符、昵称、弱别名只负责把“可能相关”的句子召回，最终是否展示提示，需要通过语义判定确认。

目标是减少两类错误：

- 误报：别人只是提到用户名字，Cue 却弹出。
- 漏报：别人没有喊全名，只用了“小张”“你们技术这边”之类表达，Cue 没有接住。

## 总体流程

```text
会前/会初生成称呼候选
  ↓
实时转写句子
  ├─ 低延迟点名路径：候选召回 → 语义判定 → 确认被问到 → 正式 Cue 生成 → 展示
  └─ 慢速上下文路径：累积上下文 → 分钟级更新 CueFrame
```

实际实现时，实时路径要尽量轻：称呼候选生成和本地匹配负责高实时召回，小模型负责快速语义判定。Cue 不需要逐句实时更新，也不应该一直生成最终文案；系统应每隔一段时间维护一份较具体的 `CueFrame`，在用户真的被问到时，把它作为提示词框架交给正式 Cue 生成器。

## 0. 称呼候选生成

会前或会初可以用 LLM 批量生成可能出现的称呼，但这些称呼不能全部视为同等可信。

输入：

- 用户主称呼：例如“张三”。
- 用户英文名、昵称、职位、团队。
- 用户主动补充的常见称呼。

输出：

```ts
type AliasCandidate = {
  text: string
  kind: "canonical" | "explicit_nickname" | "generated_alias" | "weak_alias" | "role_based"
  prior: number
  source: "user" | "llm" | "system"
}
```

示例：

- “张三”：`canonical`，高先验。
- “San”：用户输入时是 `explicit_nickname`，中高先验。
- “小张”：`weak_alias`，低先验。
- “张工”：`role_based` 或 `weak_alias`，需要上下文确认。

LLM 生成的称呼默认只作为候选，不应直接强触发。用户亲自输入或确认过的称呼，先验可以更高。

称呼候选生成使用 `deepseek-v4-pro`。第一版只保留原文称呼、弱别名和角色称呼，不做拼音化召回。

## 1. 候选召回

候选召回不是最终判断，只是为了避免每一句话都进入昂贵的语义判定。

召回信号包括：

- 命中主称呼：例如“张三”。
- 命中用户主动配置的昵称：例如“三哥”“San”。
- 命中弱别名：例如“小张”“张工”。
- 命中第二人称问题：例如“你怎么看”“你来讲一下”。
- 命中用户负责领域：例如用户负责技术，对话出现“技术这边谁来确认”。

候选召回可以用规则、轻量文本归一化和别名表完成，但召回结果只能产生候选事件，不能直接弹出 Cue。

实现上可以先用正则、别名字典或分词器处理。候选列表较大时，可以使用 trie 或 Aho-Corasick 这类多模式匹配结构；第一版别名规模很小，普通正则和归一化字符串匹配就够用。

## 2. 语义判定

语义判定模块接收候选句子和上下文，判断“这句话是否在直接问用户”。

输入应包含：

- 用户画像：主称呼、别名、弱别名、角色、负责领域。
- 当前句子：刚刚转写出的候选句。
- 上下文窗口：最近若干轮对话。
- 参会人信息：如果能获得，包含其他人的姓名、姓氏、角色。

输出结构建议：

```ts
type AddressingDecision = {
  addressedToUser: boolean
  confidence: number
  addresseeType: "direct_name" | "nickname" | "weak_alias" | "role_context" | "not_user"
  reason: string
  question: string | null
}
```

## 3. 置信度融合

最终置信度由多个信号共同决定，而不是由名字本身决定。

建议权重：

- 称呼先验：主称呼最高，明确昵称次之，弱别名较低。
- 提问意图：是否真的是在询问、指派或要求回应。
- 指向性：是否有“你”“你们这边”“来讲一下”等直接指向。
- 上下文相关性：问题是否落在用户负责领域。
- 歧义惩罚：会议里是否有其他同姓、同角色或同昵称的人。

示例阈值：

- `confidence >= 0.78`：展示 Cue。
- `0.55 <= confidence < 0.78`：进入短暂候选状态，等待下一句或补充上下文。
- `confidence < 0.55`：保持静默。

这些数值只是初始参数，后续需要用测试样本调整。

## 4. “张三 / 小张”的处理

假设用户叫“张三”：

- “张三，你怎么看？”：强触发，直接进入语义判定，通常高置信。
- “小张，你们技术这边能接吗？”：弱别名候选，需要结合用户是否负责技术、会议里是否还有其他姓张的人判断。
- “小张刚才说得对”：只是提及，不是直接提问，应保持静默。
- “小张和老王谁来排期？”：可能涉及用户，但不是明确点名，需要等待后续或降低置信。

这意味着“小张”不是“张三”的简单别名，而是一个低先验的候选称呼。

## 5. 实现建议

第一版可以采用五层实现：

1. 称呼候选生成：会前生成别名和弱称呼，保存先验权重。
2. 本地候选召回：TypeScript 规则函数，负责低成本筛选。
3. 语义判定：调用轻量模型或统一 LLM 接口，返回结构化 JSON。
4. CueFrame 更新：用 `deepseek-v4-pro` 按分钟级节奏维护一份较具体的上下文框架。
5. 正式 Cue 生成：确认用户被问到后，使用 `deepseek-v4-flash` 结合 CueFrame、当前问题和最近几轮原文生成最终提示。
6. 决策器：根据置信度、冷却时间、最近一次触发状态决定是否展示。

这样既保留语义能力，又不会让每一句转写都触发大模型调用。被点名的概率很低，所以实时路径应该服务“是否需要展示”，而 CueFrame 可以用较低频率维护宏观结构。

## 6. 模型分工

Demo 阶段统一使用 DeepSeek API：

- 称呼候选生成：`deepseek-v4-pro`。会前或会初调用，用于生成更完整的称呼、弱别名和角色称呼。
- 语义判定：`deepseek-v4-flash`。候选点名出现后快速输出结构化判断。
- CueFrame 更新：`deepseek-v4-pro`。分钟级后台任务，允许更慢，但需要更稳的宏观结构理解。
- 正式 Cue 生成：`deepseek-v4-flash`。确认被问到后快速生成最终展示文案。

核心分工是：CueFrame 可以慢，正式 Cue 必须快。

## 7. CueFrame

CueFrame 不是最终展示文案，也不需要是强一致的会议状态。它更像一个滚动上下文草稿：每次更新时，把新增上下文和上一版 CueFrame 一起交给 LLM，让模型增量改写。它可以每隔一分钟以上更新一次，也可以在明显话题切换时更新。它的作用是让正式 Cue 生成器无需从长 transcript 里重新理解会议，而是直接基于一份结构化、带宏观脉络的提示词框架生成最终提示。

```ts
type CueFrame = {
  version: number
  timeframe: string
  macroTopic: string
  currentQuestion: string | null
  recentDecisions: string[]
  openIssues: string[]
  userRelevantFacts: string[]
  likelyQuestionsToUser: string[]
  suggestedAnswerAngles: string[]
  sourceTurnIds: string[]
  updatedAt: number
}
```

更新输入建议：

```ts
type CueFrameUpdateInput = {
  previousFrame: CueFrame | null
  newTurns: TranscriptTurn[]
  userProfile: UserProfile
  updatedAt: number
}
```

CueFrame 更新不需要每次回看完整会议，只需要基于上一版 frame 和新增 transcript 做合并、修正和替换。偶尔不稳定可以接受，因为它只作为正式 Cue 的提示词框架；真正展示前还会结合当前点名句和最近几轮原文。

CueFrame 应该保持稳定结构，例如：

```text
时间范围：过去约 1-2 分钟
宏观话题：{当前主要议题}
关键变化：{刚刚发生的决策/分歧/转向}
开放问题：{还没解决的问题}
与你相关：{可能会问到用户的部分}
可回应角度：{用户被问到时可接的话}
```

这个结构会按较低频率吸收新的对话变化，保留会议的宏观脉络。正式 Cue 生成时再结合当前点名句和最近几轮原文，输出一句真正展示给用户的提示。

推荐策略：

- 会前/会初生成称呼候选，保证实时路径足够快。
- 平时用 `deepseek-v4-pro` 按分钟级更新 CueFrame。
- 候选点名出现时，快速语义检测是否真的在问用户。
- 语义检测为真：使用 `deepseek-v4-flash` 基于 CueFrame + 当前点名句 + 最近几轮原文生成正式 Cue。
- 语义检测为假：保持静默，CueFrame 继续在后台更新。

这比完全当场理解上下文更快，也比持续生成最终 Cue 更省成本。

后续需要维护一组点名检测样本，包括：

- 明确点名。
- 弱别名点名。

每个样本都应标注期望结果：展示、候选、静默。
